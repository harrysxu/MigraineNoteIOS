## 问题背景

在国际化实施后，默认标签的 `displayName` 可能会因设备语言不同而存储不同的值，导致 iCloud 同步时出现不一致。

## 问题场景

```
设备 A（中文）：创建标签 {labelKey: "nausea", displayName: "恶心"}
        ↓ iCloud 同步
设备 B（英文）：创建标签 {labelKey: "nausea", displayName: "Nausea"}
        ↓ 去重逻辑执行
结果：保留一条记录（假设保留"恶心"），但设备 B 的用户看到的是中文 ❌
```

## 解决方案

### 1. 添加 `localizedDisplayName` 计算属性

在 `CustomLabelConfig` 中添加计算属性，动态获取本地化名称：

```swift
var localizedDisplayName: String {
    if isDefault && !labelKey.isEmpty {
        // 默认标签：根据当前语言动态本地化
        let locKey = "label.\(category).\(labelKey)"
        return String(localized: String.LocalizationValue(locKey))
    } else {
        // 自定义标签：返回用户输入的原始值
        return displayName
    }
}
```

### 2. 视图层使用

所有使用标签的地方，应该使用 `localizedDisplayName` 而不是 `displayName`：

```swift
// ❌ 错误用法
Text(label.displayName)

// ✅ 正确用法
Text(label.localizedDisplayName)
```

### 3. 工作原理

#### 默认标签
- **存储**：`displayName` 可能是任何语言（取决于首次创建时的设备语言）
- **显示**：`localizedDisplayName` 始终返回当前系统语言的翻译
- **同步**：即使 iCloud 同步了不同语言的 displayName，显示时也会正确本地化

#### 自定义标签
- **存储**：`displayName` 是用户输入的原始文本
- **显示**：`localizedDisplayName` 返回原始文本（不本地化）
- **同步**：用户输入的文本在所有设备上保持一致

## 数据流示例

### 场景：两个设备初始化默认标签

```
时间线：

T1: 设备 A（中文）首次启动
    - 创建默认标签：{labelKey: "nausea", displayName: "恶心", isDefault: true}
    - 显示给用户："恶心" ✅

T2: 设备 B（英文）首次启动（尚未同步）
    - 创建默认标签：{labelKey: "nausea", displayName: "Nausea", isDefault: true}
    - 显示给用户："Nausea" ✅

T3: iCloud 同步完成
    - 设备 A 看到设备 B 的标签
    - 设备 B 看到设备 A 的标签
    - LabelManager.deduplicateLabels() 执行

T4: 去重后
    - 保留最早创建的记录（假设是设备 A 的）
    - 两个设备都有：{labelKey: "nausea", displayName: "恶心", isDefault: true}

T5: 显示结果
    - 设备 A（中文）：localizedDisplayName → "恶心" ✅
    - 设备 B（英文）：localizedDisplayName → "Nausea" ✅
      （虽然 displayName 是"恶心"，但动态获取了英文翻译）
```

## 使用 labelKey 的重要性

`labelKey` 是标签的**唯一标识符**，必须：

1. **保持英文**：便于跨语言识别
2. **全局唯一**：在同一 category 下不重复
3. **稳定不变**：一旦定义就不应修改

例如：
```swift
labelKey: "nausea"           // ✅ 好的 labelKey（英文、简短、清晰）
labelKey: "恶心"             // ❌ 不好（中文、不利于代码维护）
labelKey: "symptom_nausea"   // ✅ 也可以（更明确）
```

## 去重逻辑

`LabelManager.deduplicateLabels()` 的工作方式：

1. 按 `(category, labelKey, subcategory)` 分组
2. 每组保留 `createdAt` 最早的记录
3. 删除同组的其他重复记录

**重要**：displayName 不参与去重判断，只有 labelKey 才是唯一标识。

## 常见问题

### Q: 为什么不直接存储多语言翻译？

A: 那样会导致：
- 数据库膨胀（每条记录需要存储3份翻译）
- 添加新语言时需要数据迁移
- 增加复杂度

当前方案：
- 翻译集中在 Localizable.xcstrings
- 数据库只存储标识符（labelKey）
- 动态获取翻译（高效且灵活）

### Q: 旧数据的 displayName 会被更新吗？

A: 不会。displayName 保持不变，但：
- 显示时使用 `localizedDisplayName`
- 默认标签会动态本地化
- 自定义标签显示原始文本

### Q: 如果用户删除了默认标签的翻译怎么办？

A: `String(localized:)` 会自动回退：
1. 尝试获取当前语言翻译
2. 失败则尝试获取源语言（zh-Hans）
3. 仍失败则返回 key 本身

所以最坏情况会显示 "label.symptom.nausea"，不会崩溃。

## 总结

✅ **解决方案完整性**
- 默认标签：动态本地化，多语言完美支持
- 自定义标签：保持原样，不受语言影响
- iCloud 同步：无冲突，数据一致

✅ **性能影响**
- `localizedDisplayName` 是计算属性，每次调用都会查询本地化
- 但查询是 O(1) 操作，性能影响可忽略
- 缓存机制由系统自动处理

✅ **维护成本**
- 简单明了的规则：默认标签用 labelKey，自定义标签用原始值
- 易于测试和验证
- 未来添加新语言只需更新 Localizable.xcstrings
