## 🔍 关键发现

### 问题：视图层使用 displayName（P0 严重）

**发现**：在录入界面和首页的标签选择器中，代码使用 `label.displayName` 来显示标签、检查选中状态和保存选中标签。

**影响**：
- ❌ 单设备切换语言后，标签显示错误语言
- ❌ 多设备不同语言时，标签选择无法正确同步
- ❌ iCloud 同步后可能出现语言混乱

**受影响的文件**：
1. `SimplifiedRecordingView.swift` - 24 处
2. `HomeView.swift` - 24 处
3. `LabelRow.swift` - 1 处（显示）
4. `MedicationPresetEditor.swift` - 1 处（显示）

## 🎯 核心设计原则（已遵循）

### 1. 分离存储与显示

```swift
// 存储层：displayName（可能是任何语言）
var displayName: String = "恶心"

// 显示层：localizedDisplayName（动态本地化）
var localizedDisplayName: String {
    if isDefault && !labelKey.isEmpty {
        return String(localized: "label.\(category).\(labelKey)")
    } else {
        return displayName
    }
}
```

### 2. 使用规范

| 场景 | 使用属性 | 说明 |
|------|---------|------|
| **UI 显示** | `localizedDisplayName` | ✅ 始终显示当前语言 |
| **UI 编辑** | `displayName` | ✅ 编辑存储值 |
| **数据查询** | `displayName` | ✅ 数据库查询 |
| **数据比较** | `labelKey` | ✅ 跨语言标识 |

## 🎓 经验教训

### 1. 视图层与数据层分离
- displayName 是存储值（历史遗留，可能不准确）
- localizedDisplayName 是显示值（动态计算，始终正确）

### 2. 测试语言无关性
- 避免硬编码语言字符串
- 使用非空检查而非具体值比较
- 确保测试在任何语言环境下都能通过

### 3. labelKey 是关键
- 必须使用稳定的英文标识符
- 所有业务逻辑基于 labelKey 而不是 displayName
- 去重、查询都依赖 labelKey

### 4. 国际化不只是翻译
- 需要考虑数据模型设计
- 需要考虑 iCloud 同步策略
- 需要考虑向前兼容性
